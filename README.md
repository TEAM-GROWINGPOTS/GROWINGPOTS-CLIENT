## 📁 폴더 구조

Next.js App Router + features/shared 2계층 구조를 기반으로 해요.

```
└── src/
    ├── app/                    # Next.js App Router — 페이지별 라우팅
    │   ├── (auth)/
    │   ├── (main)/
    │   └── (onboarding)/
    ├── features/               # 페이지·기능별 도메인 컴포넌트
    │   ├── onboarding/
    │   ├── home/
    │   ├── card-view/
    │   └── node-view/
    └── shared/                 # 전 레이어에서 쓰는 범용 공통
        ├── apis/
        ├── components/
        ├── constants/
        ├── hooks/
        ├── libs/
        ├── styles/
        ├── types/
        └── utils/
```

## 🌿 GitHub 전략

### 브랜치 전략

GitHub Flow 기반으로 운영하되, 배포 안정성을 위해 `develop` 브랜치를 통합 브랜치로 추가해요.

| 브랜치    | 역할                                                 |
| --------- | ---------------------------------------------------- |
| `main`    | 배포 가능한 상태를 유지하는 최종 브랜치              |
| `develop` | 기능 브랜치들이 모이는 통합 브랜치. 모든 PR은 여기로 |

> **Merge 방식:** Squash Merge를 기본으로 해요. 커밋 히스토리가 깔끔하게 유지돼요.

### 브랜치 네이밍

`type/feature-name/#issue-number` 형식으로 관리해요. (kebab-case)

```
init/project-setup/#1
feat/login-page/#12
fix/sidebar-item/#34
refactor/api-module/#56
docs/readme-update/#78
```

## 🔍 코드 리뷰

그로잉팟은 신뢰 머지를 하지 않는 것을 원칙으로 해요.  
코드를 작성하는 것으로 개발이 끝나는 게 아니라 리뷰를 거쳐 merge되는 순간까지가 개발이라고 생각해요.  
그래서 코드 리뷰는 단순히 실수를 걸러내는 절차가 아니라 서로의 코드를 통해 배우고 지식을 나누며 함께 성장하는 과정이에요.  
아무리 급해도 리뷰 없이 merge하지 않고 리뷰를 통해 팀 전체가 코드에 대한 이해와 책임을 함께 나눠요.

### Merge 조건

- ✅ **최소 2명 Approve** 필요
- ✅ 반드시 **PR을 통해** merge (직접 push 금지)

### 리뷰에서 중요하게 보는 것

- 아키텍처 / 폴더 구조 일관성
- 예외 처리 / 에러 핸들링
- 타입 안정성 (TypeScript), API 계약 준수
- 접근성 / UX (로딩·에러 상태 등)
- 네이밍 / 가독성

> PR이 올라오면 몰아서 한 번에가 아닌, 올라오는 즉시 분산 리뷰해요.

## 💻 코딩 컨벤션

<details>
<summary><b>🧩 컴포넌트</b></summary>
<br/>

**네이밍**

- 리액트 컴포넌트 → `PascalCase` (예: `Header`)
- 유틸 / 함수 / 변수 → `camelCase`

**TSX**

- 의미 없는 `div` 남발 지양 → 시맨틱 태그 우선 (`section`, `header`, `main` 등)
- 최상단 래퍼가 의미 없으면 Fragment(`<>...</>`) 사용
- children이 없는 컴포넌트는 self closing: `<EmptyState />`

**이벤트 핸들러**

- 이벤트 핸들러 함수는 반드시 `handle` prefix

```ts
(handleSubmit, handleResetClick);
```

- 일반 함수에는 `handle` 사용 안 해요

**Export**

- `default export` 대신 `named export` 사용

</details>

<details>
<summary><b>🗂️ 폴더명</b></summary>
<br/>

- `kebab-case` 사용 (파일명도 동일)
- 소문자로 시작
- 복수형은 뒤에 `s` 붙이기

</details>

<details>
<summary><b>⛓️ 타입</b></summary>
<br/>

- 타입 / 인터페이스 이름은 `PascalCase`
- 기본적으로 `type`보다 `interface` 우선
- Props 타입은 반드시 `Props` 접미사

```ts
interface CardProps { ... }
interface HeaderProps { ... }
```

- 유니언·튜플·리터럴은 `type` 사용

```ts
type ButtonVariant = 'primary' | 'secondary';
```

</details>

<details>
<summary><b>🔧 함수</b></summary>
<br/>

**네이밍 prefix 통일**

| Prefix          | 의미                              |
| --------------- | --------------------------------- |
| `get`           | 값을 가져와 반환                  |
| `create`        | 새 값 생성                        |
| `check`         | 조건 검사                         |
| `convert`       | 형태 변환                         |
| `add` / `minus` | 더하거나 빼기                     |
| `filter`        | 필터링 결과 반환                  |
| `is`            | boolean 반환 (예: `isEmailValid`) |

- 화살표 함수를 우선 사용해요
- `key`에 랜덤 값 금지 — 동적 리스트는 고유 `id` 사용

</details>

<details>
<summary><b>🔁 메소드</b></summary>
<br/>

- 배열 복사 시 스프레드 연산자(`...`) 사용: `const copies = [...originals]`
- `for`보다 `forEach` / `map` 우선 사용
- 구조 분해 할당 사용 — `props.xxx` 반복을 줄이고, 함수 시그니처만 봐도 어떤 값을 쓰는지 드러나요

```ts
function checkIsUser({ userName, userBirth }: UserDataProps) { ... }
```

</details>
